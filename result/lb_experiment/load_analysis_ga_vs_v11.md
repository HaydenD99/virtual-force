# 无人机负载：当前方案 vs 之前方案 对比分析

## 一、负载定义与度量的变化

### 1. 之前方案 (v11 — 仅用户数/回程)

- **负载指标**：`load_index[l] = max( user_count[l]/K, throughput[l]/backhaul )`
- **JFI**：基于 `load_index`（或等价于对「每 UAV 用户数」的公平性）
- **过载/轻载**：`load_index > avg×1.3` 为过载，`< avg×0.75` 为轻载
- **含义**：只关心「每个 UAV 服务了多少用户、占了多少回程」，**不区分**用户是否还同时被地面 AP 覆盖。

### 2. 当前方案 (地面感知 — Ground-Aware)

- **负载指标**：
  - **effective_load[l]**（依赖度加权）：对每个被 UAV l 服务的用户 k，贡献为  
    `dependency = uav_beta / (ground_beta_sum + uav_beta)`  
    即「该用户对 UAV 的依赖程度」。
  - **load_index**：`max( effective_load/K, throughput/backhaul )`
- **JFI**：基于 **effective_load**（有效负载公平性）
- **过载/轻载**：基于上述 load_index（effective_load 参与）
- **含义**：
  - 用户同时被强地面 AP 覆盖 → dependency 低 → 对 UAV 有效负载贡献小
  - 用户几乎只靠该 UAV → dependency 高 → 贡献大  
  因此「负载」反映的是 **UAV 实际承担了多少「离不开自己」的用户**，而不是单纯人数。

## 二、同一部署下负载数值的差异（典型种子）

以 **Seed 42** 同一 V6 部署为例：

| 指标 | 之前 (v11) | 当前 (地面感知) |
|------|------------|-----------------|
| Users/UAV | [15, 5, 18, 11, 9, 14] | 同左 |
| JFI (或 JFI_uc) | 0.890 | 0.889 |
| **有效负载 Eff load** | （未使用） | [7.20, 1.79, 5.04, 6.93, 5.84, 5.21] |
| **JFI_eff** | — | **0.900** |

- 用户数分布：[15, 5, 18, 11, 9, 14]，最大 18、最小 5，差异大。
- 有效负载：最大 7.20、最小 1.79，**相对差异更大**（因为 18 个用户里很多有地面覆盖，有效负载约 7.2；5 个用户的 UAV 可能多为「唯一主力」，人均依赖高）。
- **初始状态**对比（同一 Seed 42）：
  - 之前：JFI ≈ 0.6448（仅用户数）
  - 当前：JFI_eff = 0.4071，JFI_uc = 0.5957  
  → 说明在热点场景下，**按「依赖度」看的负载更不均衡**（effective_load 分布更陡），更能暴露「谁在真正扛重担」。

## 三、优化目标与决策差异

| 维度 | 之前 (v11) | 当前 (地面感知) |
|------|------------|-----------------|
| 优化目标 | min_rate − λ·max(0, 0.92−**JFI_user**) | min_rate − λ·max(0, 0.92−**JFI_eff**) |
| Phase2 触发 | JFI (用户数) < 0.92 | JFI_eff (有效负载) < 0.92 |
| 过载判定 | 按 user_count/throughput | 按 effective_load/throughput |
| 扰动策略 | 过载→远离质心；轻载→靠拢过载区 | 过载+地面覆盖好→**大移**；过载+地面差→**小移**；轻载→优先去**覆盖空洞** |

因此：
- **之前**：更偏向「用户数/回程」均衡，可能把很多「有地面兜底」的用户也算成 UAV 负载。
- **当前**：更偏向「有效负载」均衡，并区分地面覆盖好坏，**在用户真正依赖 UAV 的地方更谨慎移动**。

## 四、测试结果中的负载表现对比（10 seeds）

### 1. 初始状态

- **之前**：只报 JFI（用户数），初始约 0.64–0.76。
- **当前**：JFI_eff 约 **0.25–0.64**，普遍**低于** JFI_uc（0.50–0.64），说明在热点下「依赖度负载」更不均衡，与设计一致。

### 2. V6 后（Phase1 相同）

- 同一部署下，当前方案多报 JFI_eff 与 JFI_uc。
- 例如 Seed 42：V6 后 JFI_eff=0.900，JFI_uc=0.889 → 有效负载略更均衡一点。
- 例如 Seed 88：V6 后 JFI_eff=0.731，JFI_uc=0.781 → **有效负载比用户数更不均衡**，Phase2 更有优化空间。

### 3. LB-BVF 后（Phase2 差异）

| 种子 | 之前 v11 (Users, JFI) | 当前 GA (Users, JFI_eff, JFI_uc) | 负载含义变化 |
|------|------------------------|-----------------------------------|--------------|
| 42 | [12,15,17,8,8,12], JFI=0.938 | [9,11,17,14,9,12], JFI_eff=0.880, JFI_uc=0.947 | 当前方案更追求 min_rate(+3.5)，用户数更均衡(0.947)，但**有效负载**仍不均(0.880)，因 17/14 用户中依赖度可能高 |
| 88 | [·], JFI=0.883 | JFI_eff=**0.977**, JFI_uc=0.895 | 当前方案显著拉高 **有效负载** JFI(+0.246)，体现「地面感知」在严重依赖失衡时的作用 |
| 71/107 | min_rate 大幅提升 | JFI_eff 略降、JFI_uc 略降 | 当前方案允许用略降 JFI_eff 换大幅 min_rate，因 composite 以 min_rate 为主 |

- **之前**：LB 后主要看「用户数 JFI」提升（如 +0.024 平均）。
- **当前**：  
  - 平均 **Δ JFI_eff = +0.040**（有效负载公平性提升更大）；  
  - 平均 **Δ JFI_uc = +0.003**（用户数均衡略好）；  
  - 平均 **Δ min_rate = +1.66**（略高于 v11 的 +1.41）。

## 五、小结：负载情况的变化

1. **负载语义**  
   - **之前**：负载 ≈ 每 UAV 用户数 + 回程占用，与地面是否覆盖无关。  
   - **当前**：负载 = **依赖度加权有效负载** + 回程，**考虑地面与 UAV 联合覆盖**；同一用户数下，地面覆盖越差，该 UAV 负载越高。

2. **不均衡暴露程度**  
   - 热点场景下 **JFI_eff 普遍低于 JFI_uc**，说明「按依赖度」看负载更不均衡，更能发现**真正过载**的 UAV（承担大量「离不开自己」的用户）。

3. **过载判定与策略**  
   - 过载/轻载基于 **effective_load**，更符合「谁在真正扛重担」。  
   - 扰动策略：**地面覆盖好的区域**过载 UAV 可大胆移动；**地面覆盖差的区域**只做小微调；轻载 UAV 优先填**覆盖空洞**。负载决策与网络实际能力一致。

4. **数值结果**  
   - 当前方案在 **有效负载 JFI** 上平均提升更大（+0.040 vs v11 的 JFI +0.024），在依赖度严重失衡的种子（如 88）上改善明显。  
   - 部分种子用略降 JFI_eff 换更高 min_rate，整体 composite 与 min_rate 略优于或持平 v11，且 **0 FAIL**，安全性不变。

**结论**：当前方案下无人机的「负载」从**单纯用户数/回程**变为**地面感知的有效负载（依赖度加权）**，更贴合空地联合覆盖的实际负担，不均衡暴露更充分，优化目标与扰动策略都围绕「谁真正依赖 UAV」展开，负载情况更合理、可解释性更强。
